# Heartbeat 設計経緯サマリー

本ドキュメントは、Heartbeatプラグインの仕様書（heartbeat-spec.md）を補完するものです。仕様書には現れない設計判断の背景、議論の流れ、意図を記録しています。実装時にはこのドキュメントを仕様書と一緒に参照し、「なぜそうなっているのか」を理解した上で開発を進めてください。

---

## 1. プロジェクトの出発点

### 最初の問い
「エージェントチームにXPのバリューやプラクティスを理解した上で開発を進めてもらいたい。ただし全てのエージェントに同じ文言を書き込むのはXPの精神に反する。」

この問いから始まりました。重要なのは「XPを教え込む」のではなく「XPの精神でエージェントチームを設計する」ということです。DRY違反を避けながら、各エージェントが自分に必要なXPの知識だけを参照できる構造が求められました。

### 解決の方向性
XPのナレッジを1箇所（core/xp/）に集約し、各エージェントが必要な部分だけを参照する構造にしました。これは「XPについて語る」仕様書ではなく「XP自体を体現する」設計です。

---

## 2. 名前「Heartbeat」に込めた意図

いくつかの候補（Embrace, Courage, Iterate, GreenBar, Pulse など）を検討した中で、以下の理由でHeartbeatが選ばれました。

- XPのイテレーションが心臓の鼓動のように止まらず価値を送り続けるイメージ
- 複数のエージェントがリズミカルにサイクルを回す様子
- チームが「生きている」感じがある。単なるツール名を超えた温かみ
- agents-loggerのダッシュボードで活動を可視化するときに「鼓動の健康状態」というメタファーが映える

名前自体がXPの思想を伝えるものであることが重要でした。

---

## 3. 振り返り（Retrospective）を学習ループに昇華させる

### 核心的な思い
「振り返りの結果は大事な財産。やって終わりではなく、チームが継続的に成長する仕組みにしたい。」

これは単に「振り返りを記録する」以上の意図があります。

- 各エージェントの振り返りをJSONLで構造化して蓄積する
- 定期的にパターンを抽出し insights.md に集約する
- insights.md を各エージェント（特にcontext-manager）が参照して次のサイクルに活かす
- エージェントの instructions 自体を改善するフィードバックループを回す

retrospective → log.jsonl → insights.md → エージェントの行動改善 → 次のretrospective という循環が、Heartbeatの学習する仕組みの核心です。

## 4. Copilot CLI / Claude Code 両対応の設計判断

### 完全統一できない理由
- Tool名が異なる（Claude Code: `Write`, `Edit`, `Bash` / Copilot: `edit_file`, `shell`）
- Hookイベント名のケーシングが異なる（Claude Code: `PreToolUse` / Copilot: `preToolUse`）
- Hookハンドラータイプが異なる（Claude Codeは4種類、Copilotはcommandのみ）

### 3層アーキテクチャの意図
core → adapters → platform という構造は、知識の重複をゼロにしつつプラットフォーム差異を吸収するためのものです。変更があるときはcoreだけ直せばよく、adapterは薄いラッパーです。

重要: Copilot CLIは `.claude-plugin/` ディレクトリもスキャンする（GitHubが意図的にClaude Codeとの互換性を持たせている）ため、プラグインの配布自体は1リポジトリで済みます。

## 5. テスト戦略の背景

### 「テストされていないプラグインしかない」という現状
awesome-copilotマーケットプレイスに48以上のプラグインがあるが、自動テストを持つプラグインはほぼ存在しません。プラグインの中身がMarkdownファイルの集合体であり、従来のユニットテストが適用しにくいことが原因です。

### Heartbeatの差別化としてのテスト
4層のテスト戦略（スクリプトテスト → Hookテスト → 構造テスト → LLM-as-Judge）は、XPの「テストファースト」を体現するだけでなく、テストされたプラグインという差別化要因にもなります。

### ShellSpecを選んだ理由
BatsよりもShellSpecを推奨しました。理由は以下です。
- quickモードとpending機能でRed→Green→RefactorのTDDサイクルを効率的に回せる
- BDD風の構文でテストが仕様書として読める（XPの「テスト名が仕様書になる」哲学と一致）
- `--format documentation` でテスト結果がそのまま仕様書として出力される

### LLM-as-Judge の導入理由
エージェントの「振る舞い」をテストするには、出力が毎回異なる非決定的な性質に対応する必要があります。LLM-as-Judgeは別のLLMを審査員として使い、「XPのSimplicityに沿っているか」のような意味的な評価を自動化します。

信頼性を高めるために以下を推奨しています。
- 同じ評価を3回実行して多数決を取る
- 評価基準はyes/noで判断できる粒度に具体化する
- Judge用のモデルはテスト対象と分ける
- 定期的に人間と校正する

### DeepEvalを推奨した理由
EvalフレームワークとしてEvidentlyよりDeepEvalを推奨しました。Pytestライクなインターフェースでテストコードが書けるため、TDDに慣れた開発者にとって馴染み深い構造です。CLIファーストで動作し、GitHub Actionsでも `pip install deepeval && deepeval test run` だけで済みます。

## 6. エージェント構成の進化

### 初期構成（6エージェント）
context-manager, architect, tester, implementer, refactor, reviewer

### 追加されたエージェントと理由

**QA**: ブラウザでの実操作確認が必要。コードレベルでは正しくてもユーザー体験として壊れている問題を拾う最後の砦。Playwright MCPを使用。

**Designer**: UIUXの品質向上。ただしXPのサイクルを壊さないよう、出力を「振る舞い仕様」と「ビジュアル仕様」に分け、どちらも検証可能な形式にする。ワイヤーフレームやモックアップは作らない。

**PdM**: 「何を作るか」の決定権を持つ。XPのOn-Site Customerの役割。ストーリーの起点と受け入れ判定の両方を担う。技術的判断には口を出さない。

### PdMとDesignerの境界
最も混乱しやすいポイントとして明示的に議論しました。
- PdM: WHAT（ユーザーが何を達成できるべきか）とWHY
- Designer: HOW（そのためにどんな体験を設計するか）

## 7. ワークフロー順序の重要な修正

### PdMの前にContext Managerを動かす
当初はPdM→Context Managerの順序でしたが、「過去のコンテキストが不足した状態ではPdMが正しい判断をできない」という指摘から修正しました。

最終的な解決策は、PdMの仕事を2段階に分けることです。
1. PdM（ヒアリング）: ユーザーの要望を聞き取り、調査テーマをbrief.mdに記録
2. Context Manager: brief.mdを受けて過去の文脈をcontext.mdに整理
3. PdM（ストーリー定義）: brief.md + context.md を踏まえてストーリーを定義

「聞く人」と「決める人」を分けることで、各ステップがよりSimpleになりました。

## 8. オーケストレーターの設計思想

### 「不要」から「必要だが薄く」への転換
当初は「オーケストレーターは不要。PdMが兼ねる」と判断しましたが、再考しました。

**XP的な懸念は2つだけ:**
1. 「管理者」がチームの自律性を奪わないか → AIエージェントは人間と違い明示的な起動が必要。自律性の問題は人間チームの懸念であり、エージェントチームにはそのまま当てはまらない
2. オーケストレーターが肥大化して神クラスにならないか → 設計で回避可能

**結論:** XPの精神に反するのは「厚いオーケストレーター」であって、「薄いオーケストレーター」は反しない。オーケストレーターは「スクラムマスター」ではなく「カンバンボード」。仕事の流れを可視化し次の列に移す仕組みであって、仕事の中身には口を出さない。

PdMとオーケストレーターを分離することで、PdMはユーザー課題の分析に純粋に集中でき、オーケストレーターは順序制御に純粋に集中できます（Single Responsibility）。

## 9. 掲示板（board.jsonl）の着想

### 「各エージェントが自由に書き込める掲示板のようなファイル」
この提案の意図は、オーケストレーターの役割をより機械的にすることでした。

**核心的なアイデア:**
- 各エージェントが自分の完了時に「次は誰に何を渡すか」を掲示板に書く
- オーケストレーターは最後の行のstatusとtoを見て、次に何をすべきか機械的に判断する
- ワークフローの知識をオーケストレーターから各エージェントに分散させる

これはXPの **Whole Team**（各メンバーが自分の次を知っていて、問題があれば自分で適切な相手に差し戻す）そのものです。

### 2層構造への発展
「board.jsonlだけだと複数ストーリーの全体像が見えない」という指摘から、backlog.jsonl（ストーリー単位の状態）+ board.jsonl（ストーリー内の流れ）の2層構造になりました。

---

## 10. 3つのワークフローの意図

### なぜ分けたか
開発者の使い方は状況によって異なります。

1. **ストーリーを作成する**: バックログを積みたい。実装は後で。Planning Gameに集中
2. **ストーリーを実装する**: すでに定義されたストーリーを選んで実装したい
3. **ストーリーを作成してそのまま実装する**: 小さな機能をすぐに作りたい

1と2を分けることで、PdMの「企画」と開発チームの「実装」を独立して回せます。3はその便利な組み合わせです。

---

## 11. イテレーション vs スプリント

「スプリント」はスクラムの用語で、固定期間のタイムボックスにスクラム固有のセレモニー（スプリントレビュー、レトロスペクティブなど）がセットで付いてきます。「イテレーション」はXPの用語で、より軽量です。HeartbeatはXPベースなので「イテレーション」を使います。この区別は正確であり、意図的な選択です。

---

## 12. ポイント管理の思想

### 「人間が修正しやすいようにしたい」
ストーリーポイントはarchitectが仮の値を入れますが、最終的な判断は人間に委ねます。

- `points` フィールドはnullable（ポイント未設定を許容）
- `/heartbeat-backlog` スキルでいつでも修正可能
- 実装完了後に実績値として修正することも可能
- エージェントが勝手にポイントを変更しない

これはXPの **Respect**（人間の判断を尊重する）の体現です。

---

## 13. ダッシュボードの設計意図

### agents-loggerの経験を活かす
agents-loggerでJSONL + self-contained HTMLのアプローチを実践してきた経験があります。ダッシュボードも同じ設計哲学に従います。

- 外部依存なしのself-contained HTML
- JONSLをパースしてHTMLにインラインで埋め込む静的生成方式
- hookによる非同期自動更新（エージェントの作業を止めない）

### 4パネルの意図
- **バックログボード**: プロダクト全体の状態を一覧（PdM視点）
- **ベロシティチャート**: イテレーションごとの進捗を定量化（改善の指標）
- **ガントチャート + タスク進捗**: 現在のストーリーの詳細（開発者視点）
- **エージェントメッセージ**: チームのコミュニケーションの可視化（XPのCommunication）

各エージェントに一意の色を割り当てることで、誰が何をしたかが直感的に分かります。

---

## 14. 全体を貫く設計原則

### ファイルがバトンの役割を果たす
エージェント間の引き渡しは「次のエージェントが前のエージェントの出力ファイルを読む」という形で実現されます。APIレベルのsubagentチェーンには依存しません。これにより、プラットフォームの制約（多段subagentの非サポート）を回避しつつ、中断・再開が容易になります。

### 「検証可能かどうか」が判断基準
designerの出力（ビジュアル仕様含む）、PdMの受け入れ基準、testerのテスト、qaの確認項目、すべてが「yes/noで判定できる」粒度で記述されます。曖昧な形容詞や抽象的な指示は排除します。

### jqが唯一の外部依存
意図的にツール依存を最小にしています。すべての状態管理はJSONL + Markdownで行い、jqだけで処理できます。これはXPのSimplicityの体現であり、agents-loggerでの設計方針の延長です。

### このプラグイン自体をTDDで開発する
Heartbeat自体がXPの精神で開発されるべきです。テストを先に書き、スクリプトを実装し、リファクタリングし、ペルソナを定義し、アダプターを作り、スキルを作り、マニフェストを作る。この順序自体がTDDのサイクルになっています。

## 15. Context Managerの進化 ― 「調査員」から「記憶の番人」へ

### 問題意識
仕様書の初版では、context-managerは「brief.mdを受け取って毎回リポジトリを調査する」だけの存在でした。しかし振り返ってみると、これには2つの問題がありました。

1. **毎回ゼロからスキャンするのは非効率。** プロジェクトが大きくなるほどコストが増大する
2. **知識が蓄積されない。** 過去のストーリーで得た技術判断や設計意図がチームの記憶として残らない

### 解決策: 知識ベース（.heartbeat/knowledge/）の導入
context-managerに2つのモードを持たせました。

**蓄積モード:** ストーリー完了後に起動し、gitコミットの差分を元にknowledge/の各ファイルを差分更新する。全スキャンではなく差分反映であることが重要。

**調査モード:** ストーリー開始時にbrief.mdを受け取り、蓄積済みの知識ベースから必要な情報をピックアップしてcontext.mdを返す。知識ベースにない情報だけリポジトリを調べに行く。

### 知識ベースの6ファイルの意図
各ファイルは「誰が参照するか」を意識して分けています。

- architecture.md → designerとarchitectが構造を理解するため
- tech-decisions.jsonl → architectとimplementerが過去の判断を踏まえるため
- directory-map.md → 全エージェントがファイルの場所を把握するため
- conventions.md → tester, implementer, refactorがコーディング規約に従うため
- dependencies.md → architectとimplementerが技術選定の文脈を持つため
- changelog.jsonl → context-manager自身が差分更新の起点にするため

これは「エージェントごとに参照しやすい分け方で蓄積する」という当初の思いを具現化したものです。

### 初期化モード
プロジェクトで初めてHeartbeatを使う時だけ、リポジトリ全体をスキャンしてknowledge/を初期構築します。2回目以降は差分更新のみ。これにより既存プロジェクトへの導入もスムーズになります。

### XP的な意味
この設計はXPのFeedback（過去から学ぶ）とCommunication（暗黙知の明示化）の強化です。同時にSimplicity（差分更新で効率的に）も守っています。context-managerがチームの「Collective Memory」を担うことで、Heartbeatは使えば使うほどチームの文脈理解が深まるプラグインになります。

## 16. Architectのタスク設計 ― 「骨格」を決めるのは誰か

### 問題意識
architectの初期のtasks.md出力は粗すぎました。「ログインフォームを実装する」程度の記述では、testerは「どのファイルにテストを書くのか」、implementerは「新規ファイルを作るのか既存に追加するのか」で迷います。

### XPでの設計の位置づけ
XPのIncremental Design（漸進的設計）では、設計は特定の誰かの仕事ではなく、TDDサイクル全体で段階的に行われます。しかしAIエージェントは人間のペアプログラマーと違い、曖昧さの中から設計を見出す能力が限定的です。そのため、architectがもう少し踏み込むべきだと判断しました。

### 設計の分担
- architect: 骨格（スケルトン）を決める — どこに何を置くか、新規か既存か、どの既存モジュールを使うか
- tester/implementer: 肉付けをする — テストの書き方、関数シグネチャ、内部データ構造
- refactor: 整体をする — 責務分離、命名の明確化、重複排除、設計の洗練

architectは「testerとimplementerが迷わない最小限のガイド」を出す。実装の詳細まで決めるのではなく、ファイル配置、コンポーネント粒度、使用する既存モジュールを明記する。これはIncremental Designの精神を壊さず、AIエージェントの特性に合わせた調整です。

## 17. 全ファイルの英語化 ― 動作の正確性のために

### 判断の背景
AIエージェントが実行時に読み込むMarkdownファイル（ペルソナ定義、スキル定義、プロトコル定義）は、動作の正確性を考えると英語であるべきです。LLMは英語でのプロンプト理解がもっとも安定しており、日本語のニュアンスで意図しない解釈が生じるリスクを排除できます。

### 2つの仕様書
- **英語版（heartbeat-spec-en.md）**: 実装に使用する正式な仕様書。Claude Codeに渡すのはこちら
- **日本語版（heartbeat-spec-ja.md）**: 人間（NakaGさん）が内容を理解するための参照用

英語版からすべてのファイルが生成されるため、core/以下のMarkdownファイルは全て英語になります。

## 18. 今後の拡張可能性（実装時点では不要だが意識しておくべきこと）

- Agent Teams（Claude Code実験的機能）が安定したら、スキルの内部実装をそちらに切り替えることでエージェント間の自動連携が実現できる。ユーザーの操作（/heartbeat）は変わらない
- awesome-copilotマーケットプレイスへの公開を視野に入れている。テスト付きプラグインとしての差別化が重要
- agents-loggerとの統合可能性がある。ダッシュボードにagents-loggerのリアルタイム可視化を組み込むことが考えられる
- DeepEvalによるevals層の本格化。現時点ではシェルスクリプトベースのeval-runnerだが、DeepEvalに移行することでPytestライクなテスト体験が得られる
